# gRPC KV Service

## О проекте

Это тестовое задание на разработку gRPC-сервиса на Java для работы с key-value хранилищем на базе **Tarantool 3.2.x**.

Сервис реализует следующий API:

- `put(key, value)` — сохраняет значение для нового ключа или перезаписывает значение для существующего ключа;
- `get(key)` — возвращает значение по ключу;
- `delete(key)` — удаляет запись по ключу;
- `range(key_since, key_to)` — возвращает gRPC stream пар ключ-значение из заданного диапазона;
- `count()` — возвращает количество записей в БД.

Проект сделан как небольшой, понятный сервис без лишней инфраструктуры. Основной акцент был на корректной реализации контракта, работе с `null` значениями и понятной структуре кода.

## Использованные технологии

- **Java 21**
- **gRPC**
- **Protocol Buffers**
- **Tarantool 3.2.x**
- **tarantool-java-sdk 1.5.0**
- **Maven**
- **Docker Compose** — для быстрого запуска Tarantool
- **SLF4J** — для логирования

## Что именно реализовано

### 1. Хранение данных в Tarantool

Данные сохраняются в space `KV` со схемой из задания:

- `key: string`
- `value: varbinary`, `is_nullable = true`

Инициализация space, индекса и пользователя выполняется в файле:

```text
/tarantool/init.lua
```

### 2. gRPC API

Контракт описан в файле:

```text
src/main/proto/kv_service.proto
```

Для `value` используется `optional bytes`, чтобы корректно различать два случая:

- значение действительно `null`;
- значение передано как пустой массив байт.

Это важно, потому что по условию задания методы `put` и `get` должны корректно работать с `null` в поле `value`.

### 3. Сервисный слой

Основная логика обработки gRPC-запросов находится в классе:

```text
dev.rockend.grpc.KvGrpcServiceImpl
```

В этом классе:

- валидируются входные параметры;
- вызывается репозиторий для работы с Tarantool;
- формируются gRPC-ответы;
- возвращаются понятные gRPC статусы (`INVALID_ARGUMENT`, `NOT_FOUND`, `INTERNAL`).

### 4. Репозиторий

Работа с Tarantool вынесена в отдельный слой:

- `KvRepository`
- `KvRepositoryImpl`

Такое разделение позволяет не смешивать gRPC-логику и код работы с БД.

## Основные решения по реализации

### Поддержка `null` значения

В проекте поддержан сценарий, когда запись существует, но `value = null`.

Поведение следующее:

- `put(key, null)` — запись сохраняется;
- `get(key)` — возвращает успешный ответ без поля `value`;
- `count()` — такая запись учитывается в общем количестве;
- `range()` — такая запись возвращается в потоке без поля `value`.

### Реализация `put`

Для сохранения используется `replace`, поэтому метод работает как upsert:

- если ключа ещё нет — запись создаётся;
- если ключ уже есть — значение перезаписывается.

Это соответствует условию задания.

### Реализация `get`

`get` возвращает:

- значение по ключу, если запись существует;
- `NOT_FOUND`, если ключ отсутствует.

### Реализация `delete`

`delete` удаляет запись по ключу.
Если запись не найдена, сервис возвращает `NOT_FOUND`.

### Реализация `count`

`count()` возвращает текущее количество записей в space `KV`.

### Реализация `range`

Метод `range` сделан как **server-side streaming** в gRPC.

Это значит:

- клиент отправляет один запрос;
- сервер возвращает поток из нескольких `RangeResponse`;
- каждая запись отправляется отдельно.

В проекте диапазон реализован в виде:

```text
[key_since, key_to)
```

То есть:

- `key_since` — включается;
- `key_to` — не включается.

Пример:

- `range("a", "z")` вернёт ключи от `a` до `z`, не включая `z`.

### Почему `range` читается батчами

Чтобы метод корректно работал на больших объёмах данных, диапазон читается из Tarantool не целиком, а частями.

В сервисе используется батчевая обработка:

- из Tarantool запрашивается ограниченное количество записей;
- каждая запись сразу отправляется клиенту через stream;
- затем запрашивается следующая часть диапазона.

Такой подход уменьшает нагрузку на память и лучше подходит под требование задания о работе с большим числом записей.

## Как запустить проект

### Что нужно заранее

Должно быть установлено:

- **JDK 21**
- **Maven**
- **Docker** и **Docker Compose**

## Шаг 1. Запустить Tarantool

В корне проекта выполнить:

```bash
docker compose up -d
```

После запуска Tarantool будет доступен на порту `3301`.

## Шаг 2. Собрать проект

В корне проекта выполнить:

```bash
mvn clean compile
```

Эта команда:

- соберёт проект;
- сгенерирует Java-классы из `.proto`;
- подготовит проект к запуску.

## Шаг 3. Запустить gRPC-сервер

Основной класс для запуска сервера:

```text
dev.rockend.server.KvServerApplication
```

Самый простой способ — запустить его из IDE.

Параметры подключения в текущей версии проекта захардкожены в коде:

- Tarantool host: `localhost`
- Tarantool port: `3301`
- user: `app`
- password: `app`
- gRPC server port: `9090`

После успешного запуска в логах появится сообщение о старте gRPC-сервера.

## Шаг 4. Запустить клиент для проверки

Основной класс клиента:

```text
dev.rockend.client.KvManualClient
```

Его также проще всего запускать из IDE после старта сервера.

Клиент выполняет базовую ручную проверку основных сценариев работы сервиса.

## Что проверяет клиент

Клиент последовательно отправляет запросы и выводит результат в консоль.

Проверяются следующие сценарии:

- `put/get` с `null value`;
- `put/get` с обычным значением;
- `delete`;
- запрос отсутствующего ключа;
- валидация пустого ключа;
- `count()`;
- `range()`.

## Пример проверки работоспособности

После запуска клиента в консоли можно увидеть примерно такую последовательность:

```text
=== PUT/GET with null value ===
Send PUT: ...
PUT completed
Send GET: ...
GET response: hasValue = false

=== PUT/GET with byte[] value ===
Send PUT: ...
PUT completed
Send GET: ...
GET response: hasValue = true, value = hello

=== DELETE ===
DELETE completed
GET after DELETE error: NOT_FOUND

count before = ...
count after put = ...
count after delete = ...

key = a, hasValue = false
key = b, hasValue = true, valueSize = 5
key = c, hasValue = true, valueSize = 0

All checks completed successfully.
```

Если клиент доходит до конца и выводит:

```text
All checks completed successfully.
```

значит основные сценарии работы сервиса отрабатывают корректно.

## Особенности проекта

### Обработка ошибок

В сервисе используются стандартные gRPC статусы:

- `INVALID_ARGUMENT` — некорректный входной параметр;
- `NOT_FOUND` — запись не найдена;
- `INTERNAL` — внутренняя ошибка сервера.

### Работа `range` на больших объёмах

Для метода `range` данные читаются батчами по primary index Tarantool. Это сделано для того, чтобы не загружать весь диапазон в память сразу.

### Индекс для поиска

Для поля `key` создан уникальный `TREE` index. Благодаря этому:

- `get` и `delete` выполняются по ключу;
- `range` можно выполнять по отсортированному диапазону ключей.

## Итог

В проекте реализован gRPC-сервис на Java, который:

- работает с Tarantool 3.2.x;
- поддерживает `put`, `get`, `delete`, `count`, `range`;
- корректно обрабатывает `null` в поле `value`;
- использует streaming для `range`;
- может быть проверен через встроенный клиент.
